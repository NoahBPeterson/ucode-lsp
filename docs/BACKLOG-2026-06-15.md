# ucode-lsp backlog — prioritized from the 2026-06-15 triage (v0.6.248)

142 still-present findings, regrouped by **shared mechanism** (so leverage is visible) and ranked
by: false-positive noise on real OpenWrt/LuCI code → soundness → docs-fixed-per-change.
Source of truth for per-doc evidence: `TRIAGE-2026-06-15.md`. Effort: S ≤ ~1 sitting, M a few, L multi-session.

## TL;DR recommended order

1. **printf/sprintf validator rewrite** (C1) — 7 docs, one regex+switch, printf is everywhere.
2. **builtin arg over-strictness** (C2) — ~12 docs, valid calls flagged as hard errors.
3. **builtin/module return-type `| null` + shapes** (C3) — ~9 docs, localized type-table edits, kills downstream null FPs.
4. **template-mode: cheap interim** (C9a) — small detect-and-skip kills *hundreds* of FPs per LuCI template.
5. **constant/export sync (nl80211 / rtnl / fs)** (C10) — 4 docs, mechanical data sync from C source.

Then the deeper clusters (JSDoc subsystem C7, injected-globals C8, parser/lexer C11) and provider polish (C14–C17).

### Why this order
- **C1–C3 + C9a + C10** are where the LSP is *loudest wrong* on code people actually run, and each is a single mechanism fixing many docs (high leverage, S–M effort, behavior verifiable against the `ucode`/`-T` interpreter and vendored C source).
- **C7 (JSDoc)** and **C8 (injected globals)** are high-value but L-effort / design-gated — schedule deliberately, not opportunistically.
- **C9b (template front-end)** and **provider polish (C14–C17)** are real but lower-severity; do once the FP noise floor is down.
